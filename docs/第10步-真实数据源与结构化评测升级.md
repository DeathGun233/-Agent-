# 第10步：真实数据源与结构化评测升级

## 本轮目标

这一步主要补齐 3 个更偏 AI 应用工程的能力：

- 接入 1 到 2 个真实数据源，替换部分 mock 工具
- 给 Agent 输出增加 schema 校验与自动重试
- 将评测从单一总分升级为多维评分

## 新增内容

### 1. 真实数据源

当前已接入 2 个公开可演示的数据源：

- `GitHub Issues`
  - 用于把公开仓库 issue 映射为客服 / 工单类任务
  - 适合演示“真实工单抓取 -> 自动分流 -> 人工审核”
- `NYC 311`
  - 用于把纽约市 311 公共服务请求映射为真实工单样本
  - 适合演示“公共服务投诉/请求 -> 优先级判断 -> 分流”

当前主要接在 `support_triage` 工作流中，通过 `input_payload.data_source` 切换。

### 2. Agent 结构化输出

新增了 3 个输出模型：

- `AnalystOutput`
- `ContentOutput`
- `ReviewOutput`

当前行为：

- 模型先返回 JSON
- 系统尝试按 schema 校验
- 如果缺字段、字段类型错误、值不合法，会自动重试
- 若多次失败，则退回 fallback 结果

同时记录：

- `retry_count`
- `validation_error`

这些信息已经能在详情页的执行日志中看到。

### 3. 多维评测

评测维度升级为：

- `status_match`
- `keyword_coverage`
- `result_completeness`
- `review_alignment`

评测结果页面现在除了总分，还会展示候选方案和基线方案的维度平均分。

## 演示建议

推荐你这样演示这一轮增强：

1. 在 `support_triage` 中填入 `github_issues` 数据源
2. 跑一次工作流，查看真实 issue 如何被映射成工单
3. 打开详情页，查看 `Analyst / Content / Reviewer` 的结构化输出日志
4. 运行一次评测，对比不同模型或 Prompt 在多维评分上的差异

## 收益

这一轮之后，项目不再只是“多 Agent 能跑”，而是更接近企业 AI 应用平台：

- 有真实数据接入
- 有结构化输出约束
- 有失败重试
- 有细粒度评测
- 更适合在 AI 应用开发岗位中展示工程能力
