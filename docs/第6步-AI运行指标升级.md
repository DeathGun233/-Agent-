# 第 6 步：AI 运行指标升级

## 这一步做了什么

这一步开始把项目从“能调用模型”升级成“能观察模型调用过程”的版本。

当前已经为每次 LLM 调用记录：

- `model`
- `system_prompt`
- `user_prompt`
- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `latency_ms`
- 是否触发 `fallback`
- 回退原因

## 为什么这一步重要

AI 应用开发项目如果只能展示“结果出来了”，说服力其实有限。

真正更像企业 AI 应用工程的是：

- 能知道哪一步调了模型
- 能知道调的是哪个模型
- 能知道消耗了多少 tokens
- 能知道耗时有多长
- 能知道这次是成功解析，还是走了 fallback

这些数据是后面做评测、优化 Prompt、比较模型、控制成本的基础。

## 这一步的落点

本次没有单独新建评测表，而是先把 AI 调用指标挂在工作流日志上。

也就是说：

- `AnalystAgent`
- `ContentAgent`
- `ReviewerAgent`

这三个会调用 LLM 的节点，都会把调用指标写进各自的执行日志。

这样做有两个好处：

- 不需要立刻引入额外数据库迁移
- 可以先快速形成“AI 可观测性”的第一版闭环

## 详情页新增内容

任务详情页现在新增了 `AI 运行指标` 区域，可以看到：

- 总调用次数
- 累计 tokens
- 平均耗时
- 本次使用到的模型
- 每个 Agent 的 Prompt 和耗时
- 是否走了 fallback

这样在演示项目时，就不只是“看结果”，还可以讲清楚：

- 哪一步是模型做的
- 模型成本和时延如何
- 回退策略有没有触发

## 当前边界

这一版还没有继续做：

- 独立评测数据库表
- Prompt 版本管理
- 多模型对比
- 运行结果排行榜
- 成本估算
- 图表化指标趋势

但这一步已经把最关键的原始观测数据打通了，后面继续做评测页和 A/B 对比就会顺很多。
